# Habit Tracker PWA

A mobile-first Progressive Web App for tracking daily habits. Built with Next.js, TypeScript, Tailwind CSS, and localStorage for fully client-side persistence.

---

## Getting Started

### Prerequisites

- Node.js 18 or higher
- npm 9 or higher

### Installation

```bash
cd habit-tracker
npm install
```

The `prepare` script runs automatically after install and generates the PWA icons in `public/icons/`.

If you need to regenerate icons manually:

```bash
npm run setup:icons
```

### Running the App

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) in your browser.

### Building for Production

```bash
npm run build
npm run start
```

## Running the Tests

### Unit Tests (with coverage)

```bash
npm run test:unit
```

Runs all Vitest tests and generates a coverage report. Coverage is measured for files in `src/lib/`. A minimum of 80% line coverage is required.

Coverage output:
- Terminal summary
- `coverage/` directory with HTML and JSON reports

### Integration / Component Tests

```bash
npm run test:integration
```

Runs the same Vitest suite without the coverage flag. Targets `tests/integration/` component tests.

### End-to-End Tests

```bash
npm run test:e2e
```

Requires the dev server to be available. Playwright will start `next dev` automatically if it is not already running. The first run also requires Playwright browsers:

```bash
npx playwright install chromium
npm run test:e2e
```

### Run All Tests

```bash
npm run test
```

## Local Persistence Structure

All data is stored in `localStorage` under three keys:

| Key | Shape | Description |
|-----|-------|-------------|
| `habit-tracker-users` | `User[]` | All registered users |
| `habit-tracker-session` | `Session \| null` | Currently logged-in user |
| `habit-tracker-habits` | `Habit[]` | All habits across all users |

**User shape:**
```json
{ "id": "uuid", "email": "...", "password": "...", "createdAt": "ISO string" }
```

**Session shape:**
```json
{ "userId": "uuid", "email": "..." }
```

**Habit shape:**
```json
{
  "id": "uuid",
  "userId": "uuid",
  "name": "Drink Water",
  "description": "Stay hydrated",
  "frequency": "daily",
  "createdAt": "ISO string",
  "completions": ["2024-06-15", "2024-06-14"]
}
```

Completions are stored as `YYYY-MM-DD` strings. Duplicate dates are rejected. Only habits belonging to the currently logged-in `userId` are rendered on the dashboard.

## PWA Implementation

The app is installable as a Progressive Web App through two mechanisms:

**manifest.json** (`public/manifest.json`): Declares the app name, icons, theme colour, display mode (`standalone`), and start URL. Linked from `<head>` in `layout.tsx`.

**Service Worker** (`public/sw.js`): Registered from a client component (`ServiceWorkerRegistration.tsx`) on first load. The service worker uses a stale-while-revalidate strategy:
- On `install`: caches the app shell URLs (`/`, `/login`, `/signup`, `/dashboard`, manifest, icons).
- On `fetch`: serves the cached version immediately while revalidating in the background. Falls back to cached `/` if the network is unavailable.

This means the app shell renders from cache after the first visit, even when offline.

**Icons** are generated by `scripts/generate-icons.js` using only Node.js built-ins (no external dependencies). They are solid-colour PNGs at 192×192 and 512×512.

## Trade-offs and Limitations

- **No real authentication**: Passwords are stored in plaintext in `localStorage`. This is intentional per the spec — no remote backend or auth service is used.
- **Single device only**: Because persistence is local, habits do not sync across devices or browsers.
- **No server-side rendering for protected routes**: Auth checks happen in `useEffect` on the client. There is a brief flash of `null` before redirect on protected routes.
- **Daily frequency only**: The spec requires only `'daily'` frequency. The frequency select is present in the form but locked to `daily`.
- **Offline support scope**: The service worker caches the app shell. Dynamic habit data (stored in localStorage) is always available because it is client-side. The offline experience is the app shell loading from cache; no network requests are made for habit data.

## Test File Map

| File | What it verifies |
|------|-----------------|
| `tests/unit/slug.test.ts` | `getHabitSlug` — lowercase, hyphenation, space collapsing, special character removal |
| `tests/unit/validators.test.ts` | `validateHabitName` — empty input, length limit, trimming |
| `tests/unit/streaks.test.ts` | `calculateCurrentStreak` — empty list, today-not-completed, consecutive days, duplicates, gaps |
| `tests/unit/habits.test.ts` | `toggleHabitCompletion` — add date, remove date, immutability, no duplicate dates |
| `tests/integration/auth-flow.test.tsx` | Signup creates session, duplicate email error, login stores session, invalid credentials error |
| `tests/integration/habit-form.test.tsx` | Validation error on empty name, habit creation and rendering, edit preserves immutable fields, delete requires confirmation, toggle updates streak display |
| `tests/e2e/app.spec.ts` | Full user flows: splash screen, auth redirects, signup, login, habit CRUD, streak, persistence across reload, logout, offline shell |
